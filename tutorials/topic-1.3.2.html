Virtual Memory: The Magic Trick

Virtual memory is one of the OS's greatest achievements. It lets each process think it has the entire address space to itself!

The Magic Warehouse Analogy

Imagine you run a warehouse with only 100 storage slots (physical memory). But you have 10 customers (processes), each thinking they have their own private warehouse of 1000 slots (virtual memory)!

How?
  • Give each customer a virtual address book (page table)
  • When they request slot #500, you look up the mapping: "Oh, that's actually in physical slot #37"
  • If they request slot #800 which isn't in physical warehouse? Fetch it from external storage (disk)!
  • If warehouse is full? Evict something you haven't used in a while
This is exactly how virtual memory works!

Why Virtual Memory?

  • Isolation - Processes can't access each other's memory
  • Simplicity - Each process sees a simple, contiguous address space
  • Protection - OS can mark pages as read-only, executable, etc.
  • Overcommit - Can run programs larger than physical RAM
  • Sharing - Multiple processes can map same physical page (shared libraries)

Paging

Virtual and physical memory are divided into fixed-size chunks:

  • Page - Fixed-size chunk of virtual memory (typically 4KB)
  • Frame - Fixed-size chunk of physical memory (same size as page)
  • Page Table - Maps virtual pages to physical frames

Page Table Entry (PTE)

Field Description
Frame Number Physical frame this page maps to
Present Bit 1 = in physical memory, 0 = on disk
Dirty Bit 1 = page modified, must write to disk before evict
Referenced Bit 1 = page accessed recently (for LRU)
Read/Write Permission bits
Execute Can execute code from this page?
User/Supervisor Can user-mode code access?

Translation Lookaside Buffer (TLB)

Problem: Every memory access requires 2 accesses (page table + actual data). That's 2× slower!

Solution: TLB - A fast cache of recent page table translations.

Memory Access with TLB

CPU wants to access virtual address 0x00403050:

1. Extract VPN = 0x403 (page 1027)

2. Check TLB (hardware cache):
   TLB lookup for VPN 1027

   TLB HIT? (90-95% of time)
   └─ Yes! → Frame = 142
      Go directly to physical address 0x8E050
      Total time: ~1-2 CPU cycles

   TLB MISS? (5-10% of time)
   └─ No → Walk page table in memory
      Page table lookup (100+ cycles)
      Update TLB with new mapping
      Access physical memory
      Total time: ~100-200 CPU cycles

This is why TLB hit rate is CRITICAL for performance!
Scenario Time (cycles) Slowdown
TLB hit 1-2 None (baseline)
TLB miss, page table in cache 10-20 10-20×
TLB miss, page table in RAM 100-200 100-200×
Page fault (page on disk) 1,000,000+ 1,000,000× (!)

Multi-Level Page Tables

Problem: A 32-bit address space with 4KB pages needs 1 million entries (4MB per process!).

Solution: Use a hierarchy of page tables, allocating only what's needed.

Two-Level Page Table (32-bit): Virtual Address: ┌────────────┬────────────┬────────────────────┐ │ L1 (10) │ L2 (10) │ Offset (12) │ └────────────┴────────────┴────────────────────┘ │ │ │ │ ▼ │ ┌──────────────┐ │ │ Page Dir │ │ │ (1 page) │ │ │ │ │ │ Entry 0 ────┼──┼────► NULL (not allocated) │ Entry 1 ────┼──┼────► NULL │ Entry 2 ────┼──┘ │ ... │ │ └──────────────┘ │ ▼ ┌──────────────┐ │ Page Table │ │ (1 page) │ │ │ │ Entry 0 → 5 │ │ Entry 1 → 12 │ │ ... │ └──────────────┘ │ ▼ Physical Frame Space saved: Only allocate page tables for used regions! Sparse address spaces: Only ~2-3 page tables instead of 1 million entries

Page Replacement Algorithms

When physical memory is full and we need to bring in a new page, which page should we evict?

1. First-In, First-Out (FIFO)

Evict the oldest page in memory.

  • Simple to implement
  • Poor performance (may evict frequently-used page)
  • Belady's anomaly (more frames → more faults, sometimes!)

2. Least Recently Used (LRU)

Evict the page that hasn't been used for the longest time.

  • Excellent performance (optimal in practice)
  • Expensive to implement exactly (need timestamp on every access)

3. Clock (Second Chance)

Approximates LRU using a reference bit. Cheap and effective!

Look at current hand position 2. If R=1, set R=0 and advance hand (second chance!) 3. If R=0, evict this page 4. Insert new page here

Demand Paging

Don't load pages until they're actually needed. Start with empty page table, fault in pages on demand.

Page Fault Handling

  1. CPU tries to access virtual address with Present=0
  2. MMU triggers page fault exception
  3. OS handles exception:
    • Find free frame (or evict one)
    • Read page from disk into frame
    • Update page table entry (set Present=1)
    • Restart the instruction
Thrashing: If page faults occur constantly (more time paging than executing), the system thrashes. Performance drops to nearly zero!

Cause: Too many processes for available memory.
Solution: Reduce multiprogramming level (kill processes), add more RAM, or improve working set.
