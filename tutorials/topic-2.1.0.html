  • Building web services - APIs, microservices, distributed systems
  • Debugging network issues - Why is my app slow? Why did the connection fail?
  • Performance optimization - Reducing latency, improving throughput
  • Security - Understanding attacks, implementing secure communication

Introduction: Why Networking Matters for Backend Engineers

You've been programming for years. You understand design patterns, distributed systems, and microservices. But recently, you've encountered mysterious issues:

  • A performance problem that only happens with certain request patterns
  • Intermittent connection failures that seem impossible to reproduce
  • Security vulnerabilities in your well-written code

The root cause? You're missing the foundational understanding of how networks actually work beneath the abstraction layers you've grown comfortable with.

The Postal System Analogy

Computer networks work like the postal service:
  • Letters = Data packets
  • Addresses = IP addresses
  • Post offices = Routers
  • Delivery routes = Network paths
  • Postal workers = Network protocols
Just like mail goes through sorting centers, your data goes through routers to reach its destination!

Network Models: OSI vs TCP/IP

Two models describe how networks are organized in layers:

OSI Layer TCP/IP Layer Function Examples
7. Application Application User applications HTTP, FTP, SMTP, DNS
6. Presentation Data format, encryption SSL/TLS, JPEG, ASCII
5. Session Session management NetBIOS, PPTP
4. Transport Transport End-to-end communication TCP, UDP
3. Network Internet Routing, addressing IP, ICMP, ARP
2. Data Link Network Access Node-to-node transfer Ethernet, Wi-Fi, MAC
1. Physical Physical medium Cables, Radio waves
Key Insight: The TCP/IP model is what's actually implemented on the Internet. The OSI model is more of a teaching tool. We'll focus on TCP/IP!

Encapsulation: Data Wrapping

As data travels down the network stack, each layer adds its own header (like addressing an envelope multiple times):

Encapsulation Process: Application Layer: ┌─────────────────────────────────────────┐ │ HTTP Request │ │ "GET /index.html HTTP/1.1" │ └─────────────────────────────────────────┘ ↓ (add TCP header) Transport Layer: ┌──────────────┬──────────────────────────┐ │ TCP Header │ HTTP Request │ │ Port: 80 │ Data │ └──────────────┴──────────────────────────┘ ↓ (add IP header) Network Layer: ┌─────────┬──────────────┬───────────────┐ │IP Header│ TCP Header │ HTTP Data │ │Src: A │ Port: 80 │ │ │Dst: B │ │ │ └─────────┴──────────────┴───────────────┘ ↓ (add Ethernet header) Data Link Layer: ┌──────────┬─────────┬──────────┬─────┬────┐ │Eth Header│IP Header│TCP Header│Data │FCS │ │MAC: X→Y │ │ │ │ │ └──────────┴─────────┴──────────┴─────┴────┘ ↓ Physical Layer: 010110010... (bits on wire) At receiver: Reverse process (decapsulation)

Packets, Frames, and Segments

  • Frame - Data Link layer unit (Ethernet frame)
  • Packet - Network layer unit (IP packet)
  • Segment - Transport layer unit (TCP segment or UDP datagram)
  • Message - Application layer data
