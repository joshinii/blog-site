Understanding How Web Applications Communicate Securely

Important HTTP Headers

Request Headers

  • Host: Target server (required in HTTP/1.1)
  • User-Agent: Client identification (browser, bot, etc.)
  • Accept: Content types client can handle (e.g., text/html, application/json)
  • Authorization: Credentials (e.g., Bearer token)
  • Cookie: Session information
  • Content-Type: Type of request body (e.g., application/json)
  • Content-Length: Size of request body in bytes

Response Headers

  • Content-Type: Type of response body
  • Content-Length: Size of response body
  • Set-Cookie: Create/update cookie
  • Cache-Control: Caching directives (e.g., max-age=3600)
  • Location: Redirect target (with 3xx status)
  • Server: Server software identification
  • Access-Control-Allow-Origin: CORS policy

Cookies and Sessions

Cookies enable stateful HTTP (which is stateless by design):

1. Server sends cookie in response:
HTTP/1.1 200 OK
Set-Cookie: session_id=abc123; HttpOnly; Secure; Path=/; Max-Age=3600

2. Browser stores cookie and includes it in future requests:
GET /profile HTTP/1.1
Host: example.com
Cookie: session_id=abc123

3. Server uses cookie to identify user/session

Cookie attributes:
- HttpOnly: Not accessible via JavaScript (prevents XSS)
- Secure: Only sent over HTTPS
- SameSite: Prevents CSRF attacks (Strict/Lax/None)
- Domain: Which domains can access the cookie
- Path: Which paths can access the cookie
- Max-Age / Expires: When cookie expires

HTTPS: Secure HTTP

HTTPS = HTTP over TLS/SSL. All data is encrypted between client and server.

TLS/SSL Encryption

TLS (Transport Layer Security) provides secure communication through:

TLS Components

  • Encryption: Data is encrypted so eavesdroppers can't read it
    • Symmetric encryption (AES) for data (fast)
    • Asymmetric encryption (RSA/ECDSA) for key exchange
  • Authentication: Verify server identity via certificates
    • Certificate Authority (CA) signs server certificates
    • Browser verifies certificate chain
    • Prevents man-in-the-middle attacks
  • Integrity: Detect if data was modified
    • HMAC (Hash-based Message Authentication Code)
    • Each message has a checksum

How TLS Encryption Works

Step 1: Client sends ClientHello
- Supported TLS versions (1.2, 1.3)
- Supported cipher suites (encryption algorithms)
- Random number (for key generation)

Step 2: Server sends ServerHello + Certificate
- Chosen TLS version and cipher
- Server's public key certificate
- Random number

Step 3: Client verifies certificate
- Check certificate is signed by trusted CA
- Check certificate is valid (not expired, correct domain)
- Check certificate is not revoked

Step 4: Key Exchange
- Client generates "pre-master secret"
- Encrypts it with server's public key
- Only server can decrypt (has private key)

Step 5: Both derive session keys
- Use pre-master secret + random numbers
- Generate symmetric encryption keys (AES)
- Much faster than asymmetric encryption

Step 6: Encrypted communication
- All data encrypted with session keys
- Each message authenticated (can't be tampered)
- Forward secrecy: Keys change frequently

Certificate Validation

What is an SSL/TLS Certificate?

A certificate is a digital document that proves a server's identity:

  • Domain name: example.com
  • Public key: Used to encrypt data
  • Issuer: Certificate Authority (CA) that verified the server
  • Validity period: Start and expiration dates
  • Signature: CA's digital signature proving authenticity
Certificate Validation Process
  1. Check domain match: Certificate domain matches requested URL
  2. Check validity period: Certificate not expired or not yet valid
  3. Check certificate chain: Verify CA signatures up to root CA
  4. Check revocation: Certificate not revoked (CRL/OCSP)

If any check fails → Browser shows warning or blocks connection

DNS Record Types

A Record:      example.com → 93.184.216.34 (IPv4)
AAAA Record:   example.com → 2606:2800:220:1:... (IPv6)
CNAME Record:  www.example.com → example.com (alias)
MX Record:     example.com → mail.example.com (email)
TXT Record:    example.com → "v=spf1 include:..." (text data)
NS Record:     example.com → ns1.example.com (nameserver)

DNS Query (UDP, port 53):
- Question: "What is www.example.com?"
- Answer: "93.184.216.34"
- Small messages → UDP perfect fit
- Fallback to TCP if response > 512 bytes

DNS Caching

DNS responses are cached at multiple levels to improve performance:

  • Browser cache: ~1 minute (very fast, per-application)
  • OS cache: ~5-10 minutes (shared across apps)
  • Router cache: ~30-60 minutes (shared across devices)
  • ISP resolver cache: Based on TTL (shared across customers)

TTL (Time To Live): DNS record specifies how long to cache (e.g., 3600 seconds = 1 hour)

HTTP Versions: Evolution

HTTP/1.1 vs HTTP/2 vs HTTP/3

Feature HTTP/1.1 HTTP/2 HTTP/3
Transport TCP TCP UDP (QUIC)
Multiplexing No (1 request per connection) Yes (many streams) Yes (many streams)
Header Compression No HPACK QPACK
Server Push No Yes Yes (improved)
Head-of-Line Blocking Yes (application + TCP) Partial (TCP only) No
Connection Setup TCP handshake TCP handshake 0-RTT (zero round-trip)
HTTP/2 Improvements:
  • Binary protocol: More efficient parsing (vs text-based HTTP/1.1)
  • Multiplexing: Multiple requests on single connection (no more 6 connection limit)
  • Header compression: Reduces overhead (headers often repeat)
  • Server push: Server can send resources before client requests them
HTTP/3 Advantages:
  • QUIC protocol: Built on UDP instead of TCP
  • Faster connection setup: 0-RTT for repeat connections
  • No head-of-line blocking: Lost packet only affects one stream
  • Better mobility: Connection survives IP address changes (switching Wi-Fi → 4G)

RESTful APIs

REST (Representational State Transfer) is an architectural style for building web APIs:

REST Principles

  • Stateless: Each request contains all information needed (no server-side session)
  • Resource-based: URLs identify resources (not actions)
    • Good: GET /users/123
    • Bad: GET /getUser?id=123
  • HTTP methods: Use verbs for actions (GET, POST, PUT, DELETE)
  • Representations: Resources can have multiple formats (JSON, XML, HTML)
  • HATEOAS: Responses include links to related resources

RESTful API Example

Create user:
POST /api/users
Content-Type: application/json

{"name": "Alice", "email": "alice@example.com"}

Response: 201 Created
Location: /api/users/123
{"id": 123, "name": "Alice", "email": "alice@example.com"}

Get user:
GET /api/users/123

Response: 200 OK
{"id": 123, "name": "Alice", "email": "alice@example.com"}

Update user:
PUT /api/users/123
{"name": "Alice Smith", "email": "alice@example.com"}

Response: 200 OK
{"id": 123, "name": "Alice Smith", "email": "alice@example.com"}

Delete user:
DELETE /api/users/123

Response: 204 No Content

Key Takeaways

  • HTTP is request-response protocol with methods (GET, POST, etc.) and status codes (200, 404, 500)
  • Headers carry metadata (Content-Type, Authorization, Cache-Control)
  • Cookies enable stateful interactions in stateless HTTP
  • HTTPS encrypts HTTP using TLS (confidentiality, authentication, integrity)
  • Certificates verify server identity; CAs sign certificates
  • DNS translates domain names to IP addresses (cached at multiple levels)
  • HTTP/2 adds multiplexing and compression; HTTP/3 uses QUIC (UDP) for faster connections
  • REST APIs use HTTP methods and resource-based URLs
