Why This Matters

Every program you write—from a simple Python script to a complex distributed system—runs on an operating system. The OS is the invisible conductor orchestrating thousands of programs simultaneously, each thinking it has the computer all to itself.

Here's what's happening right now as you read this:
  • Your browser is running in a process, protected from other programs by virtual memory
  • The OS scheduler gave your browser CPU time for a few milliseconds, then switched to your music player, then back—so fast you don't even notice
  • Your keystrokes travel through system calls from user space to kernel space
  • Files you're downloading? The OS is caching them in RAM and scheduling disk I/O in the background

Real Impact: A senior engineer at Netflix optimized their edge servers by understanding OS scheduling. Result? 30% improvement in request latency by reducing context switches. This tutorial teaches you those same principles!

1. Operating System Overview

What IS an Operating System?

The Factory Manager Analogy

Think of the OS as a factory manager. You have:
  • Workers (CPU cores) who do the actual work
  • Product lines (processes) competing for worker time
  • Warehouse (memory) with limited space
  • Storage facility (disk) for overflow
  • The manager (OS) deciding who works when, preventing chaos

Without the OS, programs would:

  • Fight over CPU time (no fair scheduling)
  • Overwrite each other's memory (no protection)
  • Corrupt files (no coordinated access)
  • Crash the entire system (no isolation)

The OS provides:

  • Abstraction - Programs see a simple interface, not hardware complexity
  • Multiplexing - Share limited resources among many programs
  • Protection - Isolate programs from each other and from the OS
  • Performance - Optimize resource usage for efficiency

System Calls: The Bridge Between Worlds

When a user program needs the OS to do something (read a file, allocate memory, create a process), it makes a system call.

How a System Call Works

// C code in user space
int fd = open("/etc/passwd", O_RDONLY);  // Line 1

What actually happens:
1. Application calls open() wrapper function
2. Wrapper loads system call number into register (e.g., EAX = 2 for open)
3. Wrapper loads arguments into other registers
4. Wrapper executes special instruction: "syscall" or "int 0x80"
5. CPU switches to kernel mode (Ring 0)
6. CPU jumps to system call handler in kernel
7. Kernel validates arguments (security check!)
8. Kernel performs operation (opens file)
9. Kernel returns result
10. CPU switches back to user mode (Ring 3)
11. Wrapper function returns to application

Cost: ~100-1000 nanoseconds (expensive!)
Key Insight: System calls are expensive because of the mode switch. This is why high-performance programs minimize system calls by batching operations!

Common System Calls (Linux)

Category System Calls Purpose
Process Control fork(), exec(), wait(), exit() Create, run, wait for, terminate processes

Memory brk(), mmap(), munmap() Allocate/deallocate memory
Communication pipe(), socket(), send(), recv() Inter-process communication
Information getpid(), getuid(), time() Get system information

OS Architectures

Monolithic Kernel (Linux, Unix)

┌───────────────────────────────────────┐ │ USER SPACE │ │ Apps Apps Apps Apps │ └─────────┬─────────────────────────────┘ │ System Calls ┌─────────▼─────────────────────────────┐ │ KERNEL SPACE │ │ ┌─────────────────────────────────┐ │ │ │ All OS services in one kernel │ │ │ │ • Process mgmt • File system │ │ │ │ • Memory mgmt • Drivers │ │ │ │ • Network • Scheduler │ │ │ └─────────────────────────────────┘ │ └───────────────────────────────────────┘

Pros: Fast (no message passing), efficient
Cons: One bug can crash entire OS, harder to maintain

Microkernel (Minix, QNX)

┌───────────────────────────────────────┐ │ USER SPACE │ │ Apps File Network Device │ │ Server Server Drivers │ └─────────┬────────┬─────────┬──────────┘ │ Message Passing │ ┌─────────▼────────▼─────────▼──────────┐ │ MICROKERNEL (minimal) │ │ • Inter-process communication │ │ • Basic scheduling │ │ • Low-level memory management │ └───────────────────────────────────────┘

Pros: More reliable, modular, easier to maintain
Cons: Slower (message passing overhead)
